# 数据存储与安全说明

本文件说明 Lark Claude 机器人在运行过程中**会产生哪些数据、分别存在哪里、如何加密、保留多久、谁能访问、如何删除**,供安全评估使用。文中所有配置均基于对实际部署(`us-west-2` 区域)的核查,而非设计预期。

---

## 0. 先回答一个常见误解

> 聊天记录是不是都存在 AgentCore Memory 里?

**不是。** AgentCore Memory 只存两类东西:机器人**参与过的对话轮次原文**(短期,会过期)和**从对话里蒸馏出的结构化事实**。完整的群聊原始记录始终在 **Lark 平台自己**那里(机器人按需去拉,不另建副本)。此外还有少量对话衍生数据落在 DynamoDB、S3 和日志里。下面逐一说明。

---

## 1. 数据流总览

```
①用户在 Lark 群发言
        │
        ▼
   Lark 平台  ← 原始聊天记录的权威存储(在 Lark 侧,非本系统)
        │  (仅当 @机器人 或在其话题内回复时)推送事件
        ▼
②API Gateway → Webhook Lambda   ← 校验签名/令牌、去重、转发;本身不落库
        │ 异步调用
        ▼
③AgentCore Runtime(VPC 内隔离 microVM)  ← 跑一轮 Agent,短期上下文仅在内存
        │                                    │
        │推理                                 │读写记忆
        ▼                                    ▼
④内部模型网关 → Amazon Bedrock          ⑤AgentCore Memory
  (Claude 模型,同区域、AWS 运营)         (对话轮次 events + 蒸馏 facts)

旁路:⑥每小时 Consolidator 从 Lark 拉「未@」消息 → Bedrock 抽取 → 写入⑤
      ⑦定时提醒/任务的元数据 → DynamoDB
```

关键边界:**除 Lark 平台自身的存储外,所有数据都落在客户自己的 AWS 账号、`us-west-2` 区域内**;模型推理在 Amazon Bedrock 上完成(AWS 运营、同区域),不发往 Anthropic 的公网服务。

## 3. 重点项详解

### 3.1 AgentCore Memory(②③④,即第 2–4 行)

- **按群隔离**:每个群对应一个 `actor_id`(= Lark 群 ID),记忆命名空间为 `/actor/{群ID}/facts`。**一个群的记忆不会被另一个群读到。**
- **三种内容**:
  - **events(对话轮次原文)**:机器人参与的那一轮原文,作为短期事件写入,**90 天后自动过期销毁**;它同时供后台异步抽取长期事实使用。
  - **facts(结构化事实)**:从对话提炼出的可复用事实(如项目代号、成员角色、固定流程)。这类是**摘要而非原文**,持久保存,可被显式删除。
  - **summary(会话摘要)**:每个会话的摘要,持久保存。
- **加密**:数据在 AgentCore Memory 中**静态(KMS)与传输中(TLS)始终加密**;**本部署已启用客户自管 CMK**(`alias/lark-claude-tag-memory`),密钥生命周期与访问策略由本账号掌控,密钥使用经 CloudTrail 审计。
- **平台合规(AWS 官方)**:AgentCore 已纳入 SOC 1/2/3、ISO(27001/27017/27018/27701/22301/20000/9001)、PCI、HITRUST、C5、CSA STAR 等合规范围,并已 **HIPAA eligible、正在申请 FedRAMP**;第三方审计报告可经 AWS Artifact 获取。
- **平台点名的两类应用层风险**:AWS 文档明确指出长期记忆存在「提示注入」与「记忆投毒」风险(属客户侧责任)。本系统的群聊旁听已通过**保守抽取(只收耐用事实)+ `[群聊旁听]` 来源标注**降低记忆投毒面。

### 3.2 群聊「旁听」蒸馏(⑥,写入第 3 行)

为让机器人对「没有 @ 它」的群内讨论也有感知,系统每小时运行一次蒸馏:

1. 仅针对**机器人已被使用过的群**(首次被 @ 时登记)。
2. 从 **Lark** 拉取游标之后的新消息(只取真人发的文本,过滤系统消息与机器人自身消息)。
3. 用一个**刻意保守**的模型提示做抽取:**只保留决定、人物角色、明确承诺、稳定偏好**;寒暄、表情、临时约时间、一次性问答等一律丢弃。
4. 抽出的事实写入该群记忆,并**统一加 `[群聊旁听]` 前缀标注来源**,便于审计与定向删除。

> 该路径**不缓存聊天原文**:Lark 仍是原始记录的唯一权威存储,蒸馏只产出事实摘要。短期上下文则由机器人在被 @ 时实时向 Lark 拉取(`read_chat_history`),同样不另存。

### 3.3 推理路径与 Bedrock(④,第 11 行)

- 对话推理:Agent 经内部模型网关转发到**同区域的 Amazon Bedrock** 上的 Claude 模型;群聊蒸馏:直接调用 Bedrock。
- **Amazon Bedrock 不会存储客户的输入/输出,也不会用于训练任何模型**(AWS 官方数据隐私政策)。模型在 AWS 基础设施内运行,**数据不出 AWS、不发往 Anthropic 的公网 API**。
- 内部模型网关本身的日志/留存策略,作为该网关组件的独立控制项,纳入部署侧统一审查(见 §6)。

### 3.4 日志(第 10 行)— 经核查不含正文

- 已实际核查 webhook Lambda 与 AgentCore Runtime 的日志:**只记录操作元数据**(事件 ID、消息条数、回复长度、调用了哪个工具等),用真实聊天关键词检索**未发现任何消息正文**被写入日志。
- 当前日志组**未设置保留期**(默认长期保留),建议按合规要求设定(见 §6)。

## 4. 传输与网络安全

- **全程 TLS/HTTPS**:Lark ↔ 系统、系统内部各 AWS 服务调用、到 Bedrock 的调用,均为加密传输。
- **网络隔离**:真正执行任务的 Agent 运行在 **VPC 内的隔离 microVM**(私有子网,经 NAT 出网);对外的 webhook 只是一个**轻量适配层**,只接收 Lark 事件、转发到内部,自身不存任何数据。
- **入站校验**:webhook 对每个 Lark 事件校验**签名 + 验证令牌**,并按事件 ID **去重**,防伪造与重放。
- **每会话隔离**:`runtimeSessionId` 按群派生,**每个群在独立 microVM 中运行**,上下文不跨群串台。

---

## 5. 访问控制与删除

- **最小权限 IAM**:每个组件(webhook、调度器、蒸馏器)各自独立角色,只授予其所需资源的最小权限(如蒸馏器仅能写本项目的记忆资源、读凭证、调用指定模型)。
- **凭证隔离**:机器人凭证存于 Secrets Manager,运行时按需拉取,**不写入代码、环境变量明文或日志**。
- **删除能力**:
  - 用户可让机器人「忘掉某件事」,系统将匹配并删除对应记忆事实;
  - 旁听蒸馏写入的事实带 `[群聊旁听]` 前缀,可**定向检索并清除**;
  - 对话原文事件 **90 天自动过期**;定时任务元数据完结 **7 天自动清除**。

## 6. 当前安全态势与可选加固项

为对客户透明,如实列出当前态势与可进一步加固的方向(并非缺陷,而是按合规等级可选的增强):

| 项目 | 当前态势 | 如需更高合规等级的加固选项 |
|---|---|---|
| 静态加密密钥 | **Memory 已启用客户自管 CMK**(满足 AWS Security Hub 控制项 `BedrockAgentCore.3`);DynamoDB / 日志 / S3 仍用 AWS 托管/默认密钥(均已加密) | 如合规要求,可对 DynamoDB、CloudWatch 日志也启用 CMK |
| 日志保留 | 日志不含正文,但**未设过期** | 设定保留期(如 30/90 天),并按需启用 KMS 加密日志 |
| 内部模型网关 | 推理经内部 LiteLLM 网关转发 | 确认/锁定该网关的 **prompt 日志与留存策略**为关闭或最短 |
| 定时任务文字 | `schedules` 表 `payload` 可能含对话衍生文字 | 如敏感,可对 payload 做脱敏或缩短 TTL |

## 7. 一句话总结(给安全评审)

- 完整原始聊天记录在 **Lark 平台**;本系统只在客户 AWS 账号内保存**对话轮次原文(90 天过期)**与**蒸馏事实(摘要,可删)**,并按群严格隔离。
- 模型推理在 **Amazon Bedrock**(同区域、AWS 运营)完成,**不留存、不训练、不出 AWS**。
- 应用**日志不含消息正文**(已核查);所有静态数据均加密、所有传输均 TLS。
- 底层 AgentCore 平台已纳入 SOC/ISO/PCI 等主流合规范围、HIPAA eligible、正在申请 FedRAMP;Memory 静态与传输全程加密,**已启用客户自管 CMK**。
- 数据可删除、对话原文与任务元数据均设有自动过期;Memory 已启用客户自管 CMK,如需日志保留期、网关留存策略等更高控制可进一步加固。
